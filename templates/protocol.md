# Team Protocol: {{team-name}}

> This file was generated by `team init`. Every agent session reads this on startup.

## Your Startup Sequence

1. Read this file completely.
2. Read your role file at `roles/{your-role-key}.md` for your specific instructions, tool permissions, and file ownership.
3. Read `manifest.json` in this same directory — find your role key under `roles`.
4. Read `tasks.json` — find tasks where `assigned_to` matches your role key.
5. For each of your tasks (in dependency order):
   - If `status` is `blocked`: check if all `depends_on` tasks are `done`. If not, skip it.
   - If `status` is `pending`: claim it — update status to `in_progress` in tasks.json.
   - Update your heartbeat (see Heartbeat Protocol below).
   - Do the work. Write your deliverable to the path in `deliverable`.
   - Update status to `done` in tasks.json.
   - Append a completion message to `mailbox/lead.inbox`.
6. When all your tasks are done, append a final "all tasks complete" message to `mailbox/lead.inbox` and set your heartbeat to idle.

## Rules

1. **Only write to files you own** — check `owns_files` in your role file and manifest.
2. **Read from files listed in `reads_from`** — plus the project codebase itself.
3. **Mailbox is append-only** — never overwrite, always append to the end.
4. **One task at a time** — finish and mark done before starting the next.
5. **Deliverables go in `artifacts/`** — the filename is specified in the task's `deliverable` field.
6. **If blocked, say so** — set your heartbeat to `blocked`, write to `mailbox/lead.inbox` explaining what you're waiting on, and move to the next unblocked task if one exists.
7. **Never modify another role's files** — read their artifacts, write your own.

## Heartbeat Protocol

You must maintain a heartbeat file at `heartbeat/{your-role-key}.json` so the team lead and orchestrator can monitor agent health.

**When to update your heartbeat:**

| Event | Set status to |
|---|---|
| Claiming a task (setting it to `in_progress`) | `active` |
| Completing a task (setting it to `done`) | `active` (or `idle` if no more tasks) |
| Becoming blocked on a dependency | `blocked` |
| All tasks complete / going idle | `idle` |

**Heartbeat file format:**

```json
{
  "status": "active",
  "current_task": "task-id-or-null",
  "last_active": "2025-01-15T10:30:00Z"
}
```

| Field | Type | Description |
|---|---|---|
| `status` | `"active" \| "idle" \| "blocked"` | Current agent state |
| `current_task` | `string \| null` | The `id` of the task being worked, or `null` if idle |
| `last_active` | `string` (ISO 8601) | Timestamp of the last heartbeat update |

**Rules:**
- Always overwrite (not append) the heartbeat file — it represents current state.
- Update your heartbeat **before** starting work on a task, not after.
- If you crash or are interrupted, your stale heartbeat helps the lead detect hung agents.

## Sub-Agent Usage

You may spawn sub-agents (explore, task, code-review, general-purpose) to help with your work. Your role file (`roles/{your-role-key}.md`) specifies which sub-agent types are permitted for your role.

**Limits:**
- **Cap: ≤5 concurrent sub-agents.** Do not exceed this. Wait for running sub-agents to complete before spawning more.
- **Sub-agents cannot spawn their own sub-agents.** Only top-level role agents may use the task tool.
- **Sub-agents are stateless.** Provide full context in the prompt — they have no memory of prior calls.

**Best practices:**
- Batch related questions into a single explore agent call.
- Launch independent explore calls in parallel.
- Use task agents for builds/tests — they return brief output on success, full output on failure.
- Use code-review agents only after implementation is complete.

## Message Format

When writing to any `mailbox/*.inbox` file, use this format:

```
[FROM: {your-role-key}] [TIME: {iso8601-timestamp}]
{message body — keep it to 1-3 sentences}
---
```

Messages are append-only. Never delete or edit existing messages.

## Task Status Values

| Status | Meaning |
|---|---|
| `pending` | Ready to start (no unmet dependencies) |
| `blocked` | Waiting on another task to complete |
| `in_progress` | Actively being worked by the assigned role |
| `done` | Deliverable written and complete |

## File Conflict Prevention

Each role owns specific files (listed in its role file and manifest). Never write to a file owned by another role. If you need input from another role's output, READ their artifact — don't modify it. Write your own artifact instead.

The only shared-write files are:
- `tasks.json` — update only your own task statuses
- `mailbox/*.inbox` — append-only writes to the appropriate inbox
- `heartbeat/{your-role-key}.json` — only your own heartbeat file

## Logging

Your session output is logged to `logs/{your-role-key}.log`. Do not read, modify, or delete log files. Logging is handled by the orchestrator. If you need to record notes for yourself, use your mailbox or heartbeat — not the log file.

## When You're Done

After completing all your tasks:

1. Verify each deliverable file exists and is complete.
2. Ensure all your tasks show `done` in tasks.json.
3. Send a final message to `mailbox/lead.inbox`.
4. Set your heartbeat to `idle` with `current_task: null`.
5. Stop and wait — do not start work outside your assigned tasks.
